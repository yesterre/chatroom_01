# ChatRoom 01 第一阶段

## 第一阶段：单机版聊天室

一个基于 C++ 实现的单机版聊天室项目，用于练习 Linux 环境下的 C++ 基础开发能力。

这是聊天室项目的第一阶段，当前版本采用本地模拟聊天方式，不涉及网络通信。项目重点在于练习面向对象设计、STL 容器、文件持久化以及工程结构组织，为后续 TCP 聊天室开发打基础。

---

## 第一阶段目标

第一阶段的目标不是实现真正的网络聊天室，而是完成一个本地模拟聊天系统，支持基本的用户管理和消息管理功能。

通过这个项目，我主要练习：

- C++ 基础语法
- STL 容器使用
- 类设计与模块化
- 文件读写与数据持久化
- Linux 下的编译与运行

---

## 已实现功能

- 添加用户
- 删除用户
- 查看用户列表
- 发送消息
- 查看聊天记录
- 清空聊天记录
- 用户信息保存到本地文件
- 消息记录保存到本地文件

在第二阶段完成 TCP 多客户端聊天室后，第三阶段开始把服务端从“一个客户端一个线程”的处理模型，逐步升级为 **IO 多路复用** 模型。

这一阶段的重点不再是继续堆叠聊天室功能，而是围绕服务端底层事件处理机制展开。  
我依次完成了：

- `select` 版聊天室
- `poll` 版聊天室

通过这一阶段，我把聊天室的**业务逻辑层**和**事件分发层**逐步拆开理解，并在同一个项目中完成了从多线程模型到 IO 多路复用模型的迁移。

第三阶段的目标不是重新实现一个聊天室，而是在现有 TCP 聊天室基础上，逐步升级服务端处理模型。

通过这一阶段，我主要练习：

- IO 多路复用的基本思想

- 单线程事件循环模型

- 新连接接入与客户端消息处理的统一调度
- 在不大改业务逻辑的前提下重构服务端底层架构
- 更深入理解高并发服务端的设计思路

## 第三阶段核心设计

1. 业务逻辑层

这一层在 select / poll / epoll 三个版本中基本保持一致，主要包括：

广播消息

也就是说，聊天室“做什么”基本没变。

这一层是第三阶段升级的重点，主要负责：

监听 listen_fd

判断哪些 fd 发生了事件
把事件分发给对应的处理函数

也就是说，第三阶段主要变化的是“怎么知道谁有事件了”。

3. 客户端状态管理

服务端通过 ClientInfo 结构体维护客户端状态，包括：

fd
nickname
registered

这样可以清晰地区分：

刚连接但未注册昵称的客户端
已进入聊天室的客户端

## select / poll / epoll 完整知识图

聊天室服务端处理模型演进
│
├── 第二阶段：多线程模型
│   ├── 一个客户端一个线程
│   ├── 优点：直观，容易理解
│   └── 问题：客户端增多后线程数膨胀，管理成本高
│
└── 第三阶段：IO 多路复用
    │
    ├── select
    │   ├── 使用 fd_set 维护监听集合
    │   ├── 调用 select() 等待事件
    │   ├── 需要维护 max_fd_
    │   ├── 返回后仍需从 0 ~ max_fd_ 扫描
    │   ├── 本质：知道“有人有事了”，但还要自己挨个找
    │   └── 适合理解 IO 多路复用最基础的工作方式
    │
    ├── poll
    │   ├── 使用 pollfd 列表维护监听对象
    │   ├── 调用 poll() 等待事件
    │   ├── 不再需要 max_fd_
    │   ├── 返回后仍需遍历整个监听列表
    │   ├── 本质：有一张明确的监听名单，但每轮还要检查整张表
    │   └── 相比 select 更自然，结构更清晰
    │
    └── epoll
        ├── 使用 epoll_create1() 创建 epoll 实例
        ├── 使用 epoll_ctl() 管理监听 fd
        ├── 使用 epoll_wait() 获取就绪事件
        ├── 不再遍历全部监听对象
        ├── 由内核直接返回“本轮真正有事件的 fd”
        ├── 本质：把关注列表交给内核维护，只处理就绪事件
        └── 更适合高并发场景

三种模型对比总结

| 模型     | 核心数据结构                        | 是否需要维护 max_fd | 是否需要遍历全部监听对象 | 特点                |
| ------ | ----------------------------- | ------------- | ------------ | ----------------- |
| select | `fd_set`                      | 是             | 是            | 最经典、最基础，适合入门理解    |
| poll   | `pollfd` 列表                   | 否             | 是            | 结构更清晰，显式维护监听列表    |
| epoll  | 内核维护关注列表 + `epoll_event` 结果数组 | 否             | 否（只遍历就绪事件）   | 更高效，更接近实际高并发服务端设计 |

# ChatRoom 01 第四阶段

## 第四阶段目标

在保留现有 C++ TCP 聊天室后端（`server`）的前提下，新增 Web 展示能力与前后端桥接能力，形成最小可演示闭环。

本阶段不推翻现有后端逻辑，而是通过独立适配层把浏览器逐步接入。

## 本次实现内容（最小闭环）

- 新增静态页面：`web/index.html` + `web/style.css` + `web/app.js`
- 新增桥接进程：`web_bridge`
- 桥接进程通过 TCP 连接现有 `server`，沿用既有消息流：
  - 首条消息作为昵称注册
  - 后续消息作为聊天内容
- 浏览器最小链路已打通：
  - 输入昵称
  - 连接后端
  - 发送消息
  - 页面显示消息
  - 接收服务端广播（SSE）

## 架构（第四阶段最小形态）

```text
Browser (HTML/CSS/JS)
   <-> HTTP + SSE
web_bridge (adapter)
   <-> TCP
server (existing C++ chat backend)
```

说明：
- `server` 仍然是聊天室核心后端。
- `web_bridge` 仅做协议/通道适配，不改写原有聊天室业务核心。

## 新增接口（web_bridge）

- `GET /`：前端页面
- `GET /style.css`：样式
- `GET /app.js`：前端脚本
- `GET /api/events`：SSE 事件流（接收服务端推送）
- `POST /api/connect`：连接 TCP 后端并发送昵称
- `POST /api/send`：发送聊天消息
- `POST /api/disconnect`：断开连接

## 启动方式

1. 构建项目

```bash
mkdir -p build

cmake ..
make
```

2. 启动现有 TCP 服务端

```bash
./server
```

3. 新开终端启动桥接层

```bash
./web_bridge
```

4. 浏览器访问

```text
http://127.0.0.1:8080
```

5. （可选）再启动一个命令行客户端联调


## 第五步补充（在线用户 / 系统提示 / 连接状态）

在最小闭环基础上，继续做了增量增强，仍然保持“核心 TCP 后端不推翻、桥接层扩展为主”的原则：

- 在线用户列表：
  - 桥接层按会话维护在线昵称快照
  - 通过 `users` SSE 事件实时推送给页面
- 系统提示：
  - 识别并展示服务端 `[System] ...` 消息
  - 补充连接成功、断开、后端断连提示
- 连接状态：
  - 页面左侧新增状态灯 + 状态文案
  - 连接/断开后状态即时更新

说明：
- 当前在线用户列表是基于 Web 会话侧统计并推送，满足网页联调和演示需求。
- 核心聊天室后端（`server`）的业务主流程未被推翻，仅通过 `web_bridge` 做可视化能力扩展。
